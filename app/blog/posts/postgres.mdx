### Query Execution Plan 

A **Query Execution Plan (QEP)** is the strategy chosen by the database optimizer to execute a SQL query.
For the same query, the database may choose different execution plans depending on the **table size**,
**available indexes**, **data distribution**, **statistics** and **join conditions**.

**<span className="text-emerald-500"> Q1. You have a table storing market price curves.</span>**

```sql
MARKET_PRICE
------------
curve_id
trade_date
delivery_date
price
```

**<span className="text-emerald-500"> Initially, a query on a table with `100k` took `100ms`. After the table grew to `10M` rows, the same 
query now takes `5 seconds`. What could be the probable reason?</span>**

```sql
SELECT * 
FROM market_price 
WHERE curve_id = 101
AND trade_date = CURRENT_DATE;
```

**<span className="text-blue-500"> A. Most Probable Reason: Table Size is Growing Daily</span>**

If the amount of data being read everyday increases gradually, the runtime of query execution is 
expected to increase with time.

Possible solutions:

1. **Add Appropriate Indexes**: If an appropriate index is not present, the optimizer may perform a 
full table scan, causing query execution time to increase with table size. For example, create a composite 
index on the `curve_id` and `trade_date` columns.

2. **Archive Historical Data**: Move old market prices that are rarely accessed to an archive table or
data lake.

3. **Partition Table**: Partition the table by `trade_date` (yearly or monthtly).
